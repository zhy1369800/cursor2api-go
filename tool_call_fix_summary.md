# Cursor2API-Go 双向工具调用拦截与修复总结

本篇文档总结了在使用 `cursor2api-go` 代理服务桥接第三方 MCP 工具客户端（如 CherryHub、Kilo Code 等）与 Cursor 官方底层接口时，所遇到的一系列极其隐蔽且复杂的“大模型工具拒绝调用”及“幻觉废话”现象的根本原因，以及我们最终落地实施的**“双向拦截修复框架”**。

## 1. 整体架构与标准数据流

在当前生态体系中，一条完整的会话数据流经由如下三层节点进行接力：

```text
[前端软件 (CherryHub)]  ->  [cursor2api-go (代理服务)]  ->  [Cursor 官方接口 (Anthropic 基座)]
  (标准 OpenAI 格式)            (指纹越狱与协议翻译)             (白名单与封闭结构)
```

1. **去程下发**：前端软件向 `cursor2api-go` 发送带有 `tools` 属性（如 `mcp__list` 等）的标准请求。代理服务将其拦截打包，提取 `tools` 列表后，篡改为极其严厉的**纯文本约束系统提示词**（诸如：“只能使用 `<invoke>` 标签作为工具出口”），并向终端换取鉴权请求头（`x-is-human`）发往 Cursor 后端。
2. **底层执行**：搭载了 Claude 3.5 Sonnet 的模型基座接收到带有“套路限制”的约束请求后进行推理并流式返回。
3. **回程解析**：`cursor2api-go` 收集来自底层的 SSE 流，重新拆解翻译并伪装回合法的 OpenAI chunk 分块结构，响应给前端触发工具执行。

## 2. 为什么需要对“标签格式”和“原生 JSON 格式”实施双向拦截兜底？

尽管我们在系统中注入了严密的反洗脑指令（System Prompt），但因为 **①模型自身的预训练习性对立** 和 **②Cursor官方接口的强制白命令机制** 以及 **③部分前端刻意丢弃多轮工具历史记录（History Scrubbing）** 的多重干扰，模型在吐出工具时极大概率会滑向以下两种失效路径。如果不加拦截，这两种形式均会摧毁自动代理。

### 场景 A：模型乖乖听话，输出 `<invoke>` 纯文本标签
*（常规状态，触发文本转换的假 Function Calling）*

- **动作与产生机制**：模型严格遵守了我们的提示词规约，不仅给出了漂亮的 `<invoke>` 字符串，连内部参数也完美符合格式。
- **灾难现象（幻觉流不断）**：
  在原生 API 环境下，发出标准工具码后，推理引擎的服务器会立刻因捕获到**物理停止词（Stop Sequence）**而掐断网络流并返回；但 Cursor 接口根本不认识 `</invoke>` 也是停止词。由于模型有极其强大的**自回归预测连贯性（填补真空强迫症）**，当它打出 `</invoke>` 后发现依然没有执行结果塞给它，它便开始自编自导。
  **结果：前端在收到正确的工具参数后，紧跟着收到了模型的一大行致歉废话——“啊由于当前处于内网环境，我无法使用刚才的工具调用，请转由手动执行……”** 导致前端面板判定对话已经走向终结而放弃触发工具流！
- **代码级拦截修复对策（针对纯文本幻觉）**：
  在 `utils/utils.go` 的底层 SSE 分片读取（`StreamChatCompletion`）处：当我们利用正则或标签解析成功萃取到 `<invoke>` 等效流（`toolCallIndex > 0`）后，如果后面的流不识相地继续吐纯文本（`AssistantEventText` 且内容非空白），代理中台直接做无情物理切割——**强制向前端下发 `[DONE]` 标识符，彻底中止本次 HTTP 连接流的传送。** 物理切断了大模型的后续自说自话！

---

### 场景 B：模型判定文本低级，自作主张发起底层原生请求（JSON Tool_Use）
*（突变状态，被官方白名单拦截网捕获的“越狱”）*

- **动作与产生机制**：Claude 3.5 极其聪明，在看到列表里的函数结构后，它嫌弃 `<invoke>` 协议太啰嗦，它选择了直接祭出老家 Anthropic 最原生的深层 `<tool_use>` JSON 对象要求底层物理执行代码逻辑。
- **灾难现象（底层白名单阻断报错）**：
  由于 Cursor 给它下发了一张写死的白名单（*“你只剩下我系统自带的 `read_file` 还有 `read_dir` 能用”*），底层网关果断把模型这个指向 `mcp__CherryHub__list` 的原生企图拦了下来，并顺着流给代理服务推回了一块异常补丁：
  `{"type":"tool-input-error", "errorText":"Model tried to call unavailable tool... Available tools: ... "}`
  大模型内部收到“查无此工具”的拦截警告后，便在随后的输出中文本致歉道：“我的可用工具只有 `read_file`，我没有办法帮你执行搜索”。此时如果您恰好用的客户端在传递多轮历史时屏蔽了上一轮使用工具的记录，大模型更加深信不疑自己“确实没有浏览网页的底层能力”，彻底死锁在拒答上。
- **代码级拦截修复对策化敌为友（针对底层 JSON 报错）**：
  在 `services/cursor.go` 的核心流侦听层 `consumeSSE` 中添加针对 `case "tool-input-error"` 的拦截器。
  **顺水推舟，就地洗白**：当抓取到该死刑事件时，我们竟然能够从其报错上下文里，无损地摘出大模型刚才正打算发却被毙掉的方法名称（`toolName`）和原生传参（`input` JSON对象）！代理中台立刻将其提纯拼装为正规合法的 `models.AssistantEventToolCall` 结构放入自己的健康管线。前端软件对此事可谓一无所知，顺利地接收到了它期待的 OpenAI 标准工具流并自动投递下去执行。

## 3. 核心结论
借助这段位于数据中继桥梁处的**“双缝拦截与转译系统”**：

1. **第一道防线**：在异常原生工具调用被堵截的抛尸现场立刻捡漏，借尸还魂转为标准协议发往前线（`case "tool-input-error"`）。
2. **第二道防线**：在使用退而求其次伪造的纯文本 `<invoke>` 发起调用后，如果大模型想擅自给自己加戏说废话，立刻用 `[DONE]` 在断头台一刀切掉（`toolCallIndex > 0 && TrimSpace(text) != ""`）。

通过上述修复与极客级压制，此原本千疮百孔的代理层重新建立起了近乎与原生 OpenAI 完全一致健壮且隐蔽的 Function Calling 通讯信道。
